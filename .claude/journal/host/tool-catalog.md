## [2026-04-20 17:50] — Fix: git-sidecar revert failed silently on NEW SKILL.md files

**Task:** User reported that installing a new Linear skill still produced the "description: Invalid input; credentials.0.authType: Invalid option" error in the admin UI even though the write-time + commit-time validation layers are in place. The SKILL.md the agent produced had `description:` missing and `authType: apiKey` (camelCase, should be snake_case `api_key`).
**Root cause:** `src/agent/git-sidecar.ts:164-168` (pre-fix) tried to revert .ax/ changes by running `git reset HEAD -- .ax/` then `git checkout -- .ax/`. That works for MODIFIED tracked files — `checkout --` restores them to HEAD content. For brand-new UNTRACKED files (every new-skill case), `git checkout -- <path>` fails silently with "pathspec did not match any file(s)" (we swallow with catch), the file stays in the working tree, and the subsequent `git add -A` on line 176 re-stages it. The rejected SKILL.md commits anyway; the host's skill-state reconciler parses it and surfaces state `invalid` with the raw Zod error — which is what the user sees.
**What I did:** Extracted the revert sequence into an exported `revertAxChanges(opts)` helper and added a third step: `git clean -fd -- .ax/` after the reset + checkout. That removes any untracked .ax/ additions so the revert is actually a revert, not a re-stage. Added `tests/agent/git-sidecar-revert.test.ts` with 4 cases in a real tmp git repo: (1) untracked SKILL.md is removed (was the bug), (2) modified-then-reset SKILL.md restored to HEAD, (3) non-.ax changes untouched, (4) no-op when nothing staged. Verified the test FAILS without the clean step (checked by temporarily removing it — case 1 and case 3 both fail with "file still exists after revert") and PASSES with it.
**Files touched:** src/agent/git-sidecar.ts, tests/agent/git-sidecar-revert.test.ts, .claude/journal/host/tool-catalog.md
**Outcome:** Success. Build clean. `tests/agent/` 433/433 pass (+4 new). Bug-fix-with-test policy satisfied — test would have caught the bug, verified against the old code.
**Notes:** Second-order lesson: when you "revert" via staged-state reset, you also have to account for files that weren't in HEAD to begin with. `git reset HEAD` removes them from the index; `git checkout --` can't restore them because there's no HEAD content to restore; `git clean -fd` is the missing third step. This is the same general class as our 2026-04-19 fix for "defensive writes need defensive reverts" — deleting an effectful pipeline without enumerating each effect. Here the effect was filesystem state, not index state, so the two commands needed to cover different strata.

## [2026-04-20 17:00] — Fix: sessionId placeholder rewrite desynced token ctx from catalogMap

**Task:** Agent running Linear skill via execute_script kept hitting `call_tool("mcp_linear_get_team"): unknown tool` even though `describe_tools(['mcp_linear_get_team'])` in the same turn returned the full schema (resultLen=661). User reported from live kind cluster logs.
**Root cause:** Two-step `sessionId` lifecycle:
  1. `server.ts:349-388` creates turnToken + registers `activeTokens.set(turnToken, {ctx: {sessionId: ORIGINAL, ...}})` — ORIGINAL still contains `:_:` placeholder (e.g. `http:dm:_:alice:t1`) because agentId hasn't been resolved yet.
  2. `server-completions.ts:765` inside `processCompletion` runs `sessionId = rewriteSessionPlaceholder(sessionId, agentId)` — replaces `:_:` with `:<agentId>:`. `catalogMap.set(REWRITTEN, ...)` at line 1429 uses the rewritten form.
  The agent's `http-ipc-client.ts:80-86` stamps `_sessionId: REWRITTEN` on every request (from stdin payload), and `ipc-server.ts:234-240` overrides `ctx.sessionId` with it → catalog lookup hits the rewritten key → `describe_tools` works. The `ax.callTool` preamble posts plain `{action, tool, args}` without the `_sessionId` stamp, so the handler reads `entry.ctx.sessionId = ORIGINAL` → `catalogMap.get(ORIGINAL)` returns undefined → "unknown tool".
**What I did:** Added `updateTurnCtx?: (updates: {sessionId?, agentId?}) => void` to `CompletionDeps`. `server.ts` wires it with `entry.ctx = {...entry.ctx, ...updates}` keyed on the current turnToken. Both rewrite sites in `server-completions.ts` (lines ~683 in-process and ~773 sandbox) now call `deps.updateTurnCtx?.({sessionId, agentId})` immediately after the placeholder rewrite. Regression test added in `tests/host/internal-ipc-route.test.ts` verifying that mutating entry.ctx between two POSTs on the same token is visible to the handler on the second POST (the exact shape of the fix).
**Files touched:** src/host/server-completions.ts, src/host/server.ts, tests/host/internal-ipc-route.test.ts, .claude/journal/host/tool-catalog.md
**Outcome:** Success. `npm run build` clean. `tests/host/internal-ipc-route.test.ts` 6/6 pass. Full host suite: 29 failures all pre-existing macOS `listen EINVAL` socket-path. Agent suite 429/429. The bug only reproduces in k8s mode with a sessionId that contains `:_:` (which is every per-user provisioner-resolved session), so unit coverage + kind-cluster smoke is the right shape.
**Notes:** Token ctx drift vs. per-turn maps is a pattern worth watching — any future per-turn map keyed on sessionId has the same footgun if the token is registered before agentId resolves.
