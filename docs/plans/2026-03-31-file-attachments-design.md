# File Attachments & Artifact Downloads Design

**Date:** 2026-03-31
**Status:** Draft

## Summary

Add support for attaching files to prompts (web chat + Slack) and downloading file/image artifacts generated by agents. Files are persisted to GCS (when configured) so they remain available when viewing old threads. The `/v1/files` endpoint generates signed GCS URLs for downloads. Local disk serving is the fallback when GCS is not configured.

## Supported File Types

**Images** (existing, now routed through GCS):
- `image/png`, `image/jpeg`, `image/gif`, `image/webp`

**Documents** (new):
- `application/pdf`
- `text/plain`, `text/csv`, `text/markdown`
- `application/json`
- `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` (xlsx)

Files are passed directly to the LLM as content blocks â€” no server-side text extraction. Modern LLMs handle PDFs, CSVs, etc. natively.

## Content Types

New `ContentBlock` variants for non-image file attachments, following the existing `image`/`image_data` pattern:

```typescript
// Persisted reference (fileId points to GCS/local storage)
| { type: 'file'; fileId: string; mimeType: string; filename: string }

// Transient base64 (never persisted to conversation history)
| { type: 'file_data'; data: string; mimeType: string; filename: string }
```

`file_data` is stripped before storage (defense-in-depth), same as `image_data` today.

## FileStore Changes

Add `filename` column to the `files` table to preserve the original filename for download `Content-Disposition` headers.

```typescript
export interface FileEntry {
  fileId: string;
  agentName: string;
  userId: string;
  mimeType: string;
  filename: string;     // NEW
  createdAt: string;
}
```

## GCS File Storage

### New Module: `src/host/gcs-file-storage.ts`

Wraps `@google-cloud/storage` SDK. Uses the same bucket/prefix config from `config.workspace`.

```typescript
export interface GcsFileStorage {
  /** Upload a file to GCS. */
  upload(fileId: string, buffer: Buffer, mimeType: string, filename: string): Promise<void>;

  /** Generate a signed URL for downloading a file (1hr TTL). */
  getSignedUrl(fileId: string, filename: string): Promise<string>;

  /** Check if a file exists in GCS. */
  exists(fileId: string): Promise<boolean>;
}
```

**GCS object layout:**
```
<prefix>files/<fileId>
```

Example: `workspace/files/abc-123.pdf`

Signed URLs include `Content-Disposition: inline; filename="original-name.pdf"` and `Content-Type` via response headers override.

## Upload Flow

### Web Chat

1. User attaches file via paperclip button or drag-and-drop (built into assistant-ui)
2. UI calls `POST /v1/files` with raw binary body + `Content-Type` header + `filename` query param
3. Server:
   - Generates fileId: `files/<uuid>.<ext>`
   - **GCS configured:** uploads to GCS directly
   - **No GCS:** writes to local user workspace (existing behavior)
   - Registers in FileStore (with filename)
4. Returns `{ fileId, mimeType, filename, size }`
5. UI includes `file`/`image` content blocks referencing the fileId in the next `/v1/chat/completions` request

### Slack

1. Slack sends message with file attachment URLs
2. `server-channels.ts` downloads the files
3. Uploads to GCS (or local), registers in FileStore
4. Passes `file`/`image` content blocks to the agent (same pipeline as web chat)

## Download Flow

`GET /v1/files/<fileId>`

1. Look up fileId in FileStore â†’ get GCS object key, mimeType, filename
2. **GCS configured:** generate signed URL (1hr TTL) â†’ return **302 redirect**
3. **No GCS:** serve file from local disk (existing behavior)

The 302 redirect is transparent to clients â€” `<img src="/v1/files/...">` works directly. Download links work. Old thread history works because fileIds are persisted and signed URLs are generated fresh on each request.

## Sandbox File Provisioning

Uploaded files must be available on the sandbox filesystem so the agent can process them with CLI tools.

### Flow

1. User uploads file â†’ goes to GCS, registered in FileStore
2. User sends message with `file`/`image` content blocks referencing fileIds
3. Completion pipeline starts â†’ sees file references in the message
4. Downloads files from GCS (or reads from local)
5. Writes files to the sandbox workspace before passing the turn to the agent
6. Agent has files on disk AND the LLM gets them as inline content blocks

### By sandbox mode

- **Subprocess:** new process per turn. Write files to workspace dir before spawning.
- **K8s session-long pods:** push files into the running sandbox via the existing workspace provisioning HTTP endpoint.

## Agent Artifact Flow

### workspace_write (immediate GCS upload)

When the agent calls `workspace_write`:

1. Host-side IPC handler receives the write
2. **GCS configured:** upload to GCS immediately (no local write)
3. **No GCS:** write to local workspace (existing behavior)
4. Register in FileStore with fileId
5. Return fileId in the IPC response so the agent can reference it in output

### Generated Images

Existing flow (image providers generate images) continues to work. `drainGeneratedImages()` produces image buffers which are:

1. Uploaded to GCS (instead of just local workspace)
2. Registered in FileStore
3. Returned as `image` content blocks with fileIds

## Web Chat UI Changes

### Composer

- Enable assistant-ui's built-in `ComposerPrimitive.AddAttachment` button
- Enable drag-and-drop file attachment
- On file select: upload to `/v1/files` immediately, get back fileId
- Include `image`/`file` content blocks in the message sent to `/v1/chat/completions`

### Assistant Message Rendering

- `image` blocks: render as `<img src="/v1/files/<fileId}">`
- `file` blocks: render as a downloadable file chip with filename; clicking opens `/v1/files/<fileId>` in new tab

### User Message Rendering

- Image attachments: show inline thumbnails
- File attachments: show file chips with filename and icon

### History

No special handling â€” fileIds persisted in conversation history, `/v1/files/<fileId>` generates fresh signed URLs on demand.

## Security Considerations

- All file paths constructed with `safePath()` (SC-SEC-004)
- GCS credentials stay in host process â€” never reach sandbox
- Uploaded files are taint-tagged as external content
- File size limit: 10MB (existing)
- MIME type validation on upload
- Signed URLs expire after 1 hour
- FileStore lookup prevents cross-user file access (agent/user scoping)
